# FUJU Frontend - Agent Instructions

This file provides guidelines for AI agents (GitHub Copilot, etc.) working on the FUJU Frontend project.

## Project Overview

- **Type**: SNS (Social Network Service) Web Frontend
- **Language**: TypeScript
- **Framework**: React 19
- **Build Tool**: Vite
- **Architecture**: Component-driven with custom hooks
- **Authentication**: OAuth2.0 (web), Session Cookies, and JWT support
- **API**: REST/HTTP (backend: https://github.com/NxTECH-studio/fuju-sns-backend/)
- **UI Pattern**: Responsive design with modern React patterns

## Core Principles

### 1. React & TypeScript Best Practices

**MANDATORY**: All code must be strictly typed and follow React best practices:

```typescript
// ✅ GOOD: Functional component with proper typing
import React, { FC, ReactNode } from 'react';

interface UserProps {
  userId: number;
  onUserUpdate?: (user: User) => void;
}

const UserProfile: FC<UserProps> = ({ userId, onUserUpdate }) => {
  return <div>{/* component content */}</div>;
};

export default UserProfile;

// ❌ BAD: Implicit any types, no props interface
const UserProfile = ({ userId, onUserUpdate }) => {
  return <div>{/* component content */}</div>;
};
```

**Rules**:
- ✅ Use `FC<Props>` for all functional components
- ✅ Define prop interfaces separately
- ✅ Use proper return types (e.g., `JSX.Element | null`)
- ✅ Avoid `any` types; use `unknown` if necessary
- ✅ Use const assertions for literal types
- ❌ NO implicit any types
- ❌ NO untyped component props
- ❌ NO unnecessary use of `React.FC` (Vite allows direct JSX imports)

### 2. Custom Hooks for Logic Separation

**Philosophy**: Separate business logic from UI components using custom hooks.

```typescript
// ✅ GOOD: Logic in custom hook
const useUser = (userId: number) => {
  const [user, setUser] = React.useState<User | null>(null);
  const [loading, setLoading] = React.useState(false);
  const [error, setError] = React.useState<Error | null>(null);

  React.useEffect(() => {
    const fetchUser = async () => {
      setLoading(true);
      try {
        const response = await fetch(`/api/users/${userId}`);
        if (!response.ok) throw new Error('Failed to fetch user');
        const data = await response.json();
        setUser(data.data);
      } catch (err) {
        setError(err instanceof Error ? err : new Error('Unknown error'));
      } finally {
        setLoading(false);
      }
    };
    fetchUser();
  }, [userId]);

  return { user, loading, error };
};

// Component using the hook
const UserProfile: FC<{ userId: number }> = ({ userId }) => {
  const { user, loading, error } = useUser(userId);
  if (loading) return <div>Loading...</div>;
  if (error) return <div>Error: {error.message}</div>;
  return <div>{user?.display_name}</div>;
};
```

**Rules**:
- ✅ Extract data fetching into custom hooks
- ✅ Use `useReducer` for complex state logic
- ✅ Type hook return values
- ✅ Handle loading and error states properly
- ❌ NO business logic in components
- ❌ NO multiple useState calls for related data (use useReducer)
- ❌ NO unhandled promises

### 3. API Integration with Backend

**Authentication Flows**:

**Web Clients (Session Cookies)**:
1. User clicks OAuth2 login
2. Redirect to backend OAuth authorization endpoint
3. Backend redirects back with session cookie
4. Frontend uses HttpOnly cookie for subsequent requests (automatic)
5. Logout invalidates session

**Mobile/JWT Flow Support** (for testing):
1. Exchange authorization code for JWT + refresh token
2. Store JWT in localStorage (or sessionStorage for web)
3. Include JWT in `Authorization: Bearer <token>` header
4. Refresh token when expired

```typescript
// ✅ GOOD: Abstracted API client with auth
interface ApiClientOptions {
  baseURL: string;
  token?: string;
}

class ApiClient {
  constructor(private options: ApiClientOptions) {}

  async request<T>(
    method: string,
    endpoint: string,
    body?: unknown
  ): Promise<T> {
    const url = `${this.options.baseURL}${endpoint}`;
    const headers: HeadersInit = {
      'Content-Type': 'application/json',
    };

    if (this.options.token) {
      headers['Authorization'] = `Bearer ${this.options.token}`;
    }

    const response = await fetch(url, {
      method,
      headers,
      credentials: 'include', // For session cookies
      body: body ? JSON.stringify(body) : undefined,
    });

    if (!response.ok) {
      throw new Error(`API error: ${response.status}`);
    }

    return response.json();
  }

  get<T>(endpoint: string): Promise<T> {
    return this.request<T>('GET', endpoint);
  }

  post<T>(endpoint: string, body: unknown): Promise<T> {
    return this.request<T>('POST', endpoint, body);
  }
}

export const apiClient = new ApiClient({
  baseURL: process.env.REACT_APP_API_URL || 'http://localhost:8080/v1',
});
```

**API Response Format** (from backend swagger.yaml):

```typescript
// ✅ Success response
interface ApiResponse<T> {
  data: T;
  meta: {
    timestamp: string;
    request_id: string;
  };
}

// ✅ Error response
interface ApiError {
  code: string; // e.g., "INVALID_REQUEST"
  message: string;
  timestamp: string;
}

// ✅ Type the response
const fetchUsers = async (): Promise<User[]> => {
  const response = await apiClient.get<ApiResponse<User[]>>('/users');
  return response.data;
};
```

**Rules**:
- ✅ Handle both session-based and JWT-based auth
- ✅ Include `credentials: 'include'` for session cookies
- ✅ Store and use JWT from Authorization header
- ✅ Handle token expiration gracefully
- ✅ Implement proper error handling for 401/403 responses
- ❌ NO hardcoded API URLs (use environment variables)
- ❌ NO exposing sensitive tokens in localStorage (use HttpOnly cookies if possible)
- ❌ NO unhandled API errors

### 6. Error Handling

```typescript
// ✅ GOOD: Error boundary with typed errors
interface ApiErrorResponse {
  code: string;
  message: string;
  timestamp: string;
}

class ErrorHandler {
  static handle(error: unknown): ApiErrorResponse {
    if (error instanceof Response) {
      // HTTP error
      return {
        code: `HTTP_${error.status}`,
        message: error.statusText,
        timestamp: new Date().toISOString(),
      };
    }
    if (error instanceof Error) {
      return {
        code: 'APP_ERROR',
        message: error.message,
        timestamp: new Date().toISOString(),
      };
    }
    return {
      code: 'UNKNOWN_ERROR',
      message: 'An unexpected error occurred',
      timestamp: new Date().toISOString(),
    };
  }
}

// Error Boundary component
interface ErrorBoundaryProps {
  children: ReactNode;
  fallback?: (error: ApiErrorResponse) => JSX.Element;
}

// ✅ Use try-catch in hooks
const useData = <T,>(fetcher: () => Promise<T>) => {
  const [data, setData] = React.useState<T | null>(null);
  const [error, setError] = React.useState<ApiErrorResponse | null>(null);

  React.useEffect(() => {
    fetcher()
      .then(setData)
      .catch((err) => {
        setError(ErrorHandler.handle(err));
      });
  }, []);

  return { data, error };
};
```

**Rules**:
- ✅ Implement Error Boundary component
- ✅ Handle and display API errors gracefully
- ✅ Log errors for debugging (without sensitive data)
- ✅ Provide user-friendly error messages
- ❌ NO silent error swallowing
- ❌ NO exposing internal errors to users
- ❌ NO unhandled promise rejections

```typescript
// ✅ GOOD: Type-safe environment access
interface EnvConfig {
  apiUrl: string;
  oauthClientId: string;
  oauthRedirectUri: string;
  environment: 'development' | 'production';
}

const getConfig = (): EnvConfig => {
  const apiUrl = import.meta.env.VITE_API_URL;
  const oauthClientId = import.meta.env.VITE_OAUTH_CLIENT_ID;
  const oauthRedirectUri = import.meta.env.VITE_OAUTH_REDIRECT_URI;
  const environment = (import.meta.env.VITE_ENVIRONMENT || 'development') as
    | 'development'
    | 'production';

  if (!apiUrl || !oauthClientId || !oauthRedirectUri) {
    throw new Error('Missing required environment variables');
  }

  return { apiUrl, oauthClientId, oauthRedirectUri, environment };
};
```

**Rules**:
- ✅ Use VITE_ prefix for Vite env variables
- ✅ Create typed config object
- ✅ Validate environment variables on app startup
- ❌ NO unvalidated environment variables
- ❌ NO sensitive secrets in code

### 8. Testing

**MANDATORY**:
- Minimum 80% code coverage for components
- All custom hooks must have tests
- Integration tests for critical user flows

**Test File Naming**:
- Component: `UserProfile.tsx` → Test: `UserProfile.test.tsx`
- Hook: `useUser.ts` → Test: `useUser.test.ts`

```typescript
// ✅ GOOD: Component test
import { render, screen } from '@testing-library/react';
import { UserProfile } from './UserProfile';

describe('UserProfile', () => {
  it('renders user profile with data', () => {
    const mockUser = {
      id: 1,
      username: 'testuser',
      display_name: 'Test User',
    };

    render(<UserProfile user={mockUser} />);
    expect(screen.getByText('Test User')).toBeInTheDocument();
  });

  it('shows loading state', () => {
    render(<UserProfile loading={true} />);
    expect(screen.getByText(/loading/i)).toBeInTheDocument();
  });
});

// ✅ GOOD: Hook test
import { renderHook, waitFor } from '@testing-library/react';
import { useUser } from './useUser';

describe('useUser', () => {
  it('fetches user data', async () => {
    const { result } = renderHook(() => useUser(1));

    await waitFor(() => {
      expect(result.current.user).toBeDefined();
    });
  });

  it('handles errors', async () => {
    // Mock API error
    const { result } = renderHook(() => useUser(999));

    await waitFor(() => {
      expect(result.current.error).toBeDefined();
    });
  });
});
```

**Rules**:
- ✅ Use React Testing Library (not Enzyme)
- ✅ Test behavior, not implementation
- ✅ Mock API calls with MSW (Mock Service Worker) or fetch mocks
- ✅ Write tests for error scenarios
- ❌ NO snapshot testing (unless for UI regressions)
- ❌ NO testing implementation details
- ❌ NO flaky tests (use deterministic test data)

## Common Anti-Patterns to Avoid

### ❌ Anti-Pattern 1: Props Drilling

```typescript
// BAD: Passing props through multiple levels
<Header>
  <Nav user={user} />
</Header>

// GOOD: Use Context API or custom hook
const useAuth = () => {
  const [user, setUser] = React.useState(null);
  return { user };
};
```

### ❌ Anti-Pattern 2: Missing Error Handling

```typescript
// BAD: No error handling
const fetchUser = async () => {
  const response = await fetch('/api/users/1');
  setUser(response.json());
};

// GOOD: Comprehensive error handling
const fetchUser = async () => {
  try {
    const response = await fetch('/api/users/1');
    if (!response.ok) throw new Error(`HTTP ${response.status}`);
    setUser(await response.json());
  } catch (err) {
    setError(ErrorHandler.handle(err));
  }
};
```

### ❌ Anti-Pattern 3: Missing Type Safety

```typescript
// BAD: Implicit any
const handleClick = (e) => {
  console.log(e.target.value);
};

// GOOD: Explicit types
const handleClick = (e: React.ChangeEvent<HTMLInputElement>) => {
  console.log(e.target.value);
};
```

### ❌ Anti-Pattern 4: Uncontrolled API Calls

```typescript
// BAD: API call in render
const User = ({ id }: { id: number }) => {
  const data = fetch(`/api/users/${id}`); // Infinite loop!
  return <div>{data}</div>;
};

// GOOD: useEffect with dependency array
const User = ({ id }: { id: number }) => {
  const [data, setData] = React.useState(null);

  React.useEffect(() => {
    fetch(`/api/users/${id}`).then(data => setData(data));
  }, [id]); // Dependency array prevents infinite loops

  return <div>{data}</div>;
};
```

## When Implementing New Features

Follow this process:

1. **Plan**:
   - Identify components affected
   - Review backend swagger.yaml for API changes
   - Sketch data flow (props, state, hooks)
   - Consider error scenarios

2. **Create Types**:
   - Add domain types (User, Post, Comment) in `src/types/domain.ts`
   - Add API types in `src/types/api.ts`

3. **Implement Hooks**:
   - Create custom hook for data fetching
   - Handle loading, error, success states
   - Add tests

4. **Implement Components**:
   - Start with presentational components (Dumb)
   - Connect with hooks (Smart)
   - Add proper error handling

5. **Add Styling**:
   - Use CSS Modules or variables
   - Ensure responsive design

6. **Test**:
   - Unit tests for components
   - Hook tests
   - Integration tests for critical flows

7. **Document**:
   - Update swagger.yaml reference in code comments
   - Add JSDoc comments for public APIs
